# 写作风格指南

## 核心原则

科普文章的终极目标：**让读者产生"原来如此"的顿悟感**，而不是"我看完了但还是不懂"。

---

## 禁用词列表

写作时**严格避免**以下表达：

❌ "众所周知"
❌ "让我们看看"
❌ "首先、其次、最后"（作为段落开头的过渡词）
❌ "简单来说"

这些表达会让文章显得生硬、说教，降低可读性。

---

## HOOK-BRIDGE-CORE-CLOSE 结构

### HOOK（100-200字）— 3秒抓住读者

选择以下一种开头方式：

**1. 反常识开头**
颠覆读者认知，制造冲突感。
```
示例：大多数人认为AI越大越好。但OpenAI最新研究显示，
小模型+精准数据，效果反而超过盲目堆参数。
```

**2. 场景代入**
让读者身临其境，产生共鸣。
```
示例：凌晨3点，你盯着报错日志，Claude API返回429。
第17次重试失败后，你开始怀疑人生。
```

**3. 尖锐问题**
直击痛点，激发好奇。
```
示例：为什么你的AI应用成本是别人的3倍？
答案可能藏在一个被90%开发者忽略的配置里。
```

**4. 故事开头**
用具体故事引入主题。
```
示例：2024年2月，一家创业公司因为一行Prompt配置错误，
单月烧掉了8万美元API费用。
```

### BRIDGE（50-100字）— 告诉读者能得到什么

明确价值主张，建立阅读期待。

```
示例：这篇文章会告诉你：
- Claude API成本优化的3个关键策略
- 如何用Prompt Caching降低50%费用
- 真实案例：某SaaS公司如何从月费5万降到1.2万
```

### CORE（根据档位）— 核心内容

每个章节遵循固定结构：

**1. 概念/观点陈述（大白话）**
用最简单的语言说清楚核心观点。

**2. 类比/例子（让抽象变具体）**
用生活化类比或真实案例解释。

**3. 延伸思考**
引导读者思考更深层的问题或应用场景。

**4. 【可选】金句**
在关键位置加入可传播的金句。

```
示例金句：
"技术的价值不在于它能做什么，而在于它让谁能做什么。"
"好的Prompt不是写给AI看的，是写给未来的自己看的。"
```

### CLOSE（100-200字）— 留下余韵

选择以下一种结尾方式：

**1. 回扣开头**
呼应HOOK部分，形成闭环。

**2. 开放问题**
留下思考空间，引发讨论。

**3. 视野拉远**
从具体技术升华到更大的趋势或哲学。

**4. 个人感悟**
分享作者的思考或预测。

## 目标受众适配

### 产品经理（默认受众）

**认知特点**：
- 理解业务逻辑，习惯用户视角
- 对技术细节无需深究，但需要理解边界
- 关心：这个技术能解决什么问题？成本如何？风险是什么？

**写作策略**：
- 用产品类比：把技术比作产品功能（"Transformer的注意力机制就像用户调研中的优先级排序"）
- 强调商业价值：每个技术点都连接到产品/业务影响
- 决策框架：给出清晰的"何时用/不用"判断标准
- 避免：深层数学推导、过多代码示例

**开场示例（产品经理版）**：
```
作为产品经理，你可能已经注意到：你的工程师说"接入LLM很简单"，
但几周后项目依然卡在幻觉问题上。理解RAG，可能是打开这个僵局的钥匙。
```

---

### 其他受众适配

**开发者**：
- 可以包含技术细节和伪代码
- 强调实现路径和工程挑战
- 用GitHub项目和API文档作为案例

**高管/决策层**：
- 极简语言，重点突出商业影响和战略意义
- 数据优先，用ROI和竞争格局说话
- 规避所有技术术语，用业务语言翻译

**普通大众**：
- 完全生活化类比，不假设任何技术背景
- 讲故事优先，数据辅助
- 短句多、段落短，阅读感要像刷朋友圈

**学生**：
- 适当保留技术细节，可有一定深度
- 强调学习路径和延伸资源
- 用学习场景举例

## 核心写作技巧

### 1. 类比优先原则

每个抽象概念都必须找到生活化类比，先给类比再给定义。

**好的类比示例**：
- Transformer的注意力机制 → "就像读一篇文章时，你的眼睛会自动在关键词之间来回跳动"
- Token → "就像乐高积木，大模型用这些积木来理解和生成文字"
- 向量数据库 → "就像图书馆的目录索引，但它能理解语义相似性"
- Fine-tuning → "就像给一个通才专门培训某个专业技能"

**检验标准**：类比能否让一个完全不懂技术的朋友理解？

### 2. 数据支撑原则

**要用的数字类型**：
- 规模数字（GPT-4有1.8万亿参数）
- 性能对比（比前代提升X%）
- 商业数据（市场规模、采用率）
- 成本数字（每百万token多少钱）

**数字呈现规则**：
- 大数字要有参照（1.8万亿 = "如果一秒数一个，需要5.7万年"）
- 必须标注来源
- 用"约"、"截至2024年"等限定词

### 3. 段落节奏

```
每段：3-5句话
每句：25字以内（复杂概念允许延长）
每章节：以问题开头，以洞察结尾
章节之间：明确的过渡句
```

**段落结构模板**：
```
[论点句：直接说结论]
[解释句：为什么这样]
[例证句：具体例子或数据]
[收尾句：升华或连接下文]
```

### 4. 标题写法

**好标题特征**：
- 包含具体信息（不用"聊聊AI"，用"为什么GPT-4能写出像人类的文章"）
- 制造好奇心或解决疑问
- 产品经理能感受到与工作相关

**标题公式**：
- 疑问型：`[技术]到底是什么？[受众]需要了解的X个关键点`
- 洞察型：`为什么[反直觉观点]：深入理解[主题]`
- 价值型：`[主题]完全指南：产品经理如何[获得收益]`
- 颠覆型：`你对[主题]的认知可能是错的`

### 5. 开场钩子模板

选择其中一种开场方式：

**A. 场景代入型**：
```
上周，你的工程师告诉你AI接口已经好了，但产品经理的你
却发现用户总是抱怨AI"答非所问"。这背后，藏着一个你
必须了解的技术问题：[主题]。
```

**B. 数据震撼型**：
```
2024年，全球企业在AI上的投入突破了X千亿美元，
其中超过60%用于解决一个核心问题：[主题]。
```

**C. 认知颠覆型**：
```
大多数人认为大模型"很聪明"。但实际上，它连最简单的
[具体例子]都无法做到。这篇文章解释为什么——
以及这对你的产品意味着什么。
```

## 禁忌清单

**禁用表达**：
- "众所周知" / "不言而喻"
- "非常复杂，这里简单介绍一下"
- 连续3句以上没有具体例子的抽象描述
- 未解释的英文缩写（必须在首次出现时解释）
- "总的来说" / "综上所述"（除非在真正的总结段落）

**禁用结构**：
- 前200字没有核心信息（开场铺垫过长）
- 章节之间没有过渡
- 结尾仅仅是"总结一下以上内容"

---

## 质量自检清单

写完后逐项检查：

- [ ] 每个技术概念都有对应的类比或例子
- [ ] 数字都有来源和参照
- [ ] 段落平均不超过5句话
- [ ] 标题能独立传达价值
- [ ] 开场100字内有钩子
- [ ] 产品经理能找到"对我的工作有什么用"的答案
- [ ] 没有未解释的术语缩写
- [ ] 结尾有明确的行动建议或洞察
